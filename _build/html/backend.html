backend.IMC(TS, P, S=None, customInterpolator=False)[source]¶

Interpolating Model Checking

As specified at S. Fulvio Rollini, “Craig Interpolation and Proof Manipulation: Theory and Applications to Model Checking,” Università della Svizzera Italiana. p. 38. available at https://verify.inf.usi.ch/sites/default/files/RolliniPhDThesis.pdf

A property to be verified is encoded as a formula \(P\) , so that the system is safe if the error states where \(¬P\) holds are not reachable from \(S\).

Verifying that the system satisfies \(P\) reduces to prove that \(P\) is an inductive invariant property:

\[S ⊨ P\qquad P ∧ T ⊨ P'\]

If (i) the initial states satisfy \(P\) and, (ii) assuming \(P\) holds, it also holds after applying the transition relation, then \(P\) holds in all reachable states. When the inductiveness of \(P\) cannot be directly proved, it might be possible to show that another formula \(\hat P\), stronger than \(P ( \hat P |= P )\), is an inductive invariant, from which \(P\) would follow as a consequence; this algorithm, which combines interpolation and bounded model checking (BMC), is based on iteratively building such a \(\hat P\).

backend.PDR(P, TS, get_currently_known_invariant=Ellipsis, strengthen=Ellipsis, lift=Ellipsis, k_init=1, k_max=inf, pd=True, inc=<function <lambda>>)[source]¶

Iterative-Deepening k-Induction with Property Direction.

As specified at D. Beyer and M. Dangl, “Software Verification with PDR: Implementation and Empirical Evaluation of the State of the Art” arXiv:1908.06271 [cs], Feb. 2020, Accessed: Mar. 05, 2022. [Online]. Available https://arxiv.org/abs/1908.06271.

Parameters

  • k_init (int) – the initial value \(≥1\) for the bound k

  • k_max (int) – an upper limit for the bound k

  • inc (Callable[[int], int]) – a function \(ℕ → ℕ\) such that \(∀n ∈ ℕ: inc(n) > n\)

  • TS (TransitionSystem) – Contains predicates defining the initial states and the transfer relation

  • P (FNode) – The safety property

  • get_currently_known_invariant – used to obtain the strongest invariant currently available via a concurrently running (external) auxiliary-invariant generator

  • pd (bool) – boolean flag pd (reminding of “property-directed”) is used to control whether failed induction checks are used to guide the algorithm towards a sufficient strengthening of the safety property P to prove correctness; if pd is set to false, the algorithm behaves exactly like standard k-induction.

  • lift –

    Given a failed attempt to prove some candidate invariant \(Q\) by induction, the function lift is used to obtain from a concrete counterexample-to-induction (CTI) state a set of CTI states described by a state predicate C. An implementation of the function \(k ∈ ℕ, \Inv ∈ ℕ × (S → 𝔹) × (S → 𝔹) × S → (S → 𝔹)\) and \(C = \lift(k, \Inv , Q, s)\), lift needs to satisfy the condition that for a CTI \(s ∈ S\) where \(S\) is the set of program states, the following holds:

    \[C(s) ∧ \left( ∀s_n ∈ S: C(s_n) ⇒ \Inv(s_n) ∧ ⋀_{i=n}^{n+k−1} (Q(s_i) ∧ T(s_i ,s_{i+1})) ⇒ ¬Q(s_{n+k}) \right)\]

    which means that the CTI s must be an element of the set of states described by the resulting predicate C and that all states in this set must be CTIs, i.e., they need to be k-predecessors of \(¬Q\)-states, or in other words, each state in the set of states described by the predicate \(C\) must reach some \(¬Q\)-state via \(k\) unrollings of the transition relation \(T\).

  • strengthen – The function strengthen: \(ℕ × (S → 𝔹) × (S → 𝔹) → (S → 𝔹)\) is used to obtain for a k-inductive invariant a stronger k-inductive invariant, i.e., its result needs to imply the input invariant, and, just like the input invariant, it must not be violated within k loop iterations and must be k-inductive.

Return type

bool | Status

Returns

True if P holds, Status.UNKNOWN if k > k_max , False otherwise.

backend.lifting_query(s, z, T, t)[source]¶

The lifting query takes the form \(s ∧ z ∧ T ∧ ¬t'\), which asks whether an \(s\)-state has a successor other than \(t\) under input assignment \(z\). Since this is not the case by construction, the query yields an unsatisfiable core that typically reveals a significantly reduced partial assignment that can replace \(s\).

As per J. Birgmeier, A. R. Bradley, and G. Weissenbacher, “Counterexample to Induction-Guided Abstraction-Refinement (CTIGAR),” p.6.
